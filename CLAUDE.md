# プロジェクト運用ルール

このプロジェクトでは、以下のルールに従って開発・コード管理を行います。

## Claude への必須指示

**コードを変更したら、必ずその都度 Git コミットを作成し、GitHub にプッシュしてください。**

- 変更のたびにコミット＆プッシュを行うことは、このプロジェクトにおける必須の作業手順です。
- Claude がコードを編集・生成した場合も例外なく適用されます。
- プッシュ後、`git log --oneline -5` でコミットが正しく記録されていることを確認してください。

---

## Git 運用ルール

1. **変更はすべて Git で管理する**
   - 新しい機能や修正、スタイル調整など、コードに変更を加えた場合は必ず Git でコミットします。

2. **こまめにコミットする**
   - 1つのコミットは一つの目的（バグ修正、機能追加、リファクタリングなど）に絞ります。
   - 変更内容が大きい場合は、意味のある小さな単位に分割してコミットします。

3. **コミットメッセージはわかりやすく書く**
   - 何を、なぜ変更したのかが簡単にわかるようにします。
   - 例: `fix: ログイン画面のバリデーションを修正`

4. **コード変更のたびに GitHub へプッシュする（必須）**
   - コードを変更したら、**毎回必ず** ローカルの変更を GitHub にプッシュします。
   - プッシュを省略しないでください。変更ごとにリモートへ反映することが必須です。
   - プッシュすることで、変更がリモートに保存され、共同作業やバックアップにもなります。

5. **プルリクエスト / マージ**
   - チームで開発する場合は、直接 `main` や `master` にプッシュするのではなく、ブランチを切ってプルリクエストを作成します。
   - レビューを受け、問題がなければマージします。

## ブランチ運用の基本（推奨）

- `main` / `master`: 常にデプロイ可能な状態を保つ
- `feature/*`: 新機能や改善のための作業ブランチ
- `fix/*`: バグ修正のための作業ブランチ
- `hotfix/*`: 緊急修正用のブランチ

## コード変更時の標準手順

コードを変更するたびに、以下の手順を必ず実施してください。

1. 変更をステージング
   ```bash
   git add <変更したファイル>
   ```
2. コミット
   ```bash
   git commit -m "fix: ユーザー登録のバリデーションを改善"
   ```
3. GitHub にプッシュ（毎回必須）
   ```bash
   git push origin <branch-name>
   ```
4. プッシュ確認
   ```bash
   git log --oneline -5
   ```
